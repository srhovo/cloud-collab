# 阶段8D：生产部署交接包与安全初始化入口

## 目标

阶段8C已经完成正式运行时、审核、设备治理、回滚、完整导出和独立管理员控制台的代码与测试。阶段8D不再增加业务能力，而是把未来需要真实平台权限的步骤固化为可重复构建、机器校验和默认零写入的交接包。

本阶段不部署EdgeOne、不填写真实环境变量、不访问真实Blob、不生成真实密钥、不修改8.2.31候选、不晋升8.3.0。

## 交接包

执行：

```bash
npm run production:handoff:build
```

生成：

```text
dist/production-handoff-v1/
├── handoff-manifest.json
├── activation-phases.json
├── 人工操作清单.md
├── config/production.env.template
├── release/production-launch-plan-v1.json
└── admin/
    ├── production-console.html
    ├── production-console.css
    └── production-console.js
```

交接包不包含普通用户`index.html`，也不包含任何真实密钥。管理员控制台逐字节复制自已验证源文件，但继续与普通用户三文件发布白名单隔离。

## EdgeOne Blob正确操作方式

Blob命名空间不需要在控制台手工创建。Makers Function或外部SDK首次调用：

```js
getStore('cloud-collab-production-v1')
getStore('cloud-collab-admin-production-v1')
```

时，平台会自动创建命名空间。EdgeOne控制台的Blob页面只用于只读查看命名空间和对象目录。

阶段8D提供外部一次性初始化器：

```bash
node scripts/run-edgeone-production-bootstrap-v1.mjs
```

默认只输出计划，真实读写均为0。真实执行必须同时满足：

```text
--execute
EDGEONE_PROJECT_ID
EDGEONE_API_TOKEN
CLOUD_PRODUCTION_BOOTSTRAP_CONFIRMATION=INITIALIZE-see-see_cz-V1
```

初始化器使用强一致读取、`onlyIfNew`不可变写入和写后强一致复核。目标对象存在且完全一致时安全重放；存在不同内容时失败关闭。

## 手动GitHub Actions

工作流：

```text
stage8d-edgeone-production-bootstrap
```

它只允许`workflow_dispatch`，不会因push或pull request自动执行。

### plan

默认操作。它：

- 不读取EdgeOne秘密；
- 不连接Blob；
- 生成交接包；
- 生成10个初始化资源的零写入计划；
- 上传脱敏证据包。

### execute

只有选择`execute`并输入精确确认词时才进入真实写入任务。还必须提前配置两个GitHub仓库秘密：

```text
EDGEONE_PROJECT_ID
EDGEONE_API_TOKEN
```

Token不得写入Issue、PR、Actions输入、聊天或仓库文件。

## 负责人未来操作路径

### 1. 获取EdgeOne项目ID和API Token

在EdgeOne Makers控制台中打开当前项目并记下项目ID。随后在Makers控制台创建API Token。API Token具有账户级操作能力，只保存一次并立即放入GitHub Secrets或密码管理器。

### 2. 写入GitHub仓库秘密

路径：

```text
GitHub仓库
→ Settings
→ Secrets and variables
→ Actions
→ New repository secret
```

分别创建：

```text
EDGEONE_PROJECT_ID
EDGEONE_API_TOKEN
```

不要把值发送到聊天。

### 3. 先运行零写入计划

路径：

```text
GitHub仓库
→ Actions
→ stage8d-edgeone-production-bootstrap
→ Run workflow
→ operation: plan
```

预期结果：

```text
operation=plan
status=ready_not_executed
resourceCount=10
realBlobReadsPerformed=0
realBlobWritesPerformed=0
realBlobDeletesPerformed=0
```

### 4. 执行一次性初始化

确认计划无误后：

```text
GitHub仓库
→ Actions
→ stage8d-edgeone-production-bootstrap
→ Run workflow
→ operation: execute
→ confirmation: INITIALIZE-see-see_cz-V1
```

预期首次结果：

```text
status=initialized
createdCount=10
productionCapabilitiesEnabled=false
stablePromotionAuthorized=false
```

精确重放预期：

```text
status=already_initialized_exact
createdCount=0
existingExactCount=10
```

### 5. 在EdgeOne只读核对Blob

路径以当前控制台实际名称为准：

```text
EdgeOne Makers
→ 当前项目
→ Blob存储
```

应看到：

```text
cloud-collab-production-v1
cloud-collab-admin-production-v1
```

控制台没有“新建Blob”按钮属于正常现象。

### 6. 配置正式环境变量

有长期HTTPS入口后：

```text
EdgeOne Makers
→ 当前项目
→ Settings
→ Environment Variables
```

复制`config/production.env.template`中的变量名。八项随机秘密必须彼此不同，至少32字节。聊天中出现过的密码不得使用。

环境变量修改只应用于后续部署，因此保存后必须重新部署`main`。

### 7. 自定义域名

路径：

```text
EdgeOne Makers
→ 当前项目
→ Domain Management
→ Add custom domain
```

含中国大陆加速区的自定义域名需要先完成ICP备案。正式公共域名关联production环境，并自动跟随生产分支最新成功部署。

### 8. 分阶段开启

严格按照`activation-phases.json`：

```text
disabled
→ bootstrap_once
→ read_sync_only
→ admin_foundation
→ ordinary_submission_small_cohort
→ ordinary_auto_approval
→ admin_operations
→ sensitive_manual_review
```

每次只进入一个新阶段，保存环境变量后重新部署，再完成该阶段验收。不得一次性把所有开关改为1。
