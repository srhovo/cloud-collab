# 阶段4A：普通价格提交协议 v1

## 本阶段目标

建立普通精确价格的服务端提交协议、白名单校验、标准化、Hash、幂等键和审核资格判断。

本阶段**不开放任何真实写入接口**，不注册设备、不签发令牌、不保存候选、不修改公共版本，也不执行自动批准。

## 原子提交结构

```json
{
  "schemaVersion": 1,
  "submissionId": "sub_01JABCDEF0123456789XYZABCD",
  "deviceId": "dev_01JABCDEF0123456789XYZABCD",
  "groupId": "group_fixture",
  "libraryId": "lib_receive_fixture",
  "dataType": "exact_price",
  "operation": "upsert",
  "basePublicVersion": 3,
  "clientCreatedAt": 1784376000000,
  "payload": {
    "serviceName": "测试服务A",
    "settleType": "round",
    "unitPrice": 110
  }
}
```

顶层和 payload 均采用严格白名单，多一个字段或少一个字段都拒绝。

## 当前开放范围

- `dataType`：仅 `exact_price`
- `operation`：仅 `upsert`
- `settleType`：`round` 或 `hour`
- `unitPrice`：大于 0、不超过 1000000、最多 3 位小数
- `serviceName`：Unicode NFKC、合并重复空格、长度 1～50、禁止控制字符
- 单次请求体：最大 64KB

## 业务键与内容 Hash

服务端必须忽略客户端自报的业务键和内容 Hash，并根据规范化数据重新计算：

- `businessKey = bk_v1_ + SHA-256(identity)`
- `contentHash = ch_v1_ + SHA-256(content)`
- 编码使用 Base64URL，不保留 `=`

算法与阶段3B公共快照完全一致。测试服务A（按局，110）的预期值：

```text
businessKey: bk_v1_ja06mv-cCqOze_uiSIK4YjKoixcrewF-NIQXmAiTyTQ
contentHash: ch_v1_8cLsoYwgKnmAXyDgNjXMkDLlD2t2JjdXyKrde_KhoqA
```

## 幂等

幂等键由服务端根据 `deviceId + submissionId` 计算：

```text
idem_v1_<SHA-256 Base64URL>
```

相同设备重复发送同一 submissionId 时，后续实现必须返回同一提交结果，不能创建第二条候选。

## 资格判断

- 与正式公共数据完全相同：`duplicate_noop`
- 新增价格且只有一个普通设备：`waiting_confirmation`
- 新增价格且两个不同设备提交相同内容：`eligible_auto_approval`
- 可信设备提交新增普通安全内容：`eligible_auto_approval`
- 已有公共价格发生任何变化：`pending_review`
- 有冲突候选或业务键异常：`pending_review`

“价格变化±10%”不属于不可变提交协议。后续如确需使用，只能放入独立、可版本化、可审计、可回滚的审核策略，并在启用前另行确认。

阶段4A无论返回何种资格：

```text
publicMutationAllowed = false
autoApprovalEnabled = false
```
