# 阶段7H：8.2.30候选发布预演与中国大陆访问入口

## 一、阶段结论

阶段7H只建立可验证的候选发布链路，不执行真实部署，不晋升稳定版，不开启正式公共写入，也不写入或删除Blob。

冻结候选身份：

```text
候选版本：8.2.30
候选标题：码单器8.2.30（公共协作发布候选版）
候选字节数：1,154,030
候选SHA-256：82bef41a655cd8528a138f7f2d7f7630b10bc391a95738704905c1e0647be89f
稳定版本：8.2.25
稳定晋升授权：false
稳定晋升执行：false
正式公共写入：false
```

## 二、入口拓扑

| 角色 | 入口 | 当前用途 | 当前状态 |
|---|---|---|---|
| 权威源 | GitHub仓库、PR、Actions | 代码、审计证据、候选构建 | 已启用 |
| 中国大陆建议主入口 | EdgeOne Pages候选预览 | 面向普通用户的候选静态入口 | 仅完成配置与预演，未创建项目、未部署 |
| 备用入口 | GitHub Pages | EdgeOne异常时的候选备用入口 | 改为独立手动授权，未触发部署 |
| 离线兜底 | 已验证单HTML | 本机直接打开并保留本地计算能力 | 候选文件已生成 |
| 可选源码镜像 | Gitee仓库 | 人工应急下载源码或候选 | 不作为当前自动发布前提 |

GitHub仓库页面和GitHub Pages不应作为中国大陆普通用户唯一入口。Wi-Fi偶发打不开、切换移动数据后恢复，更可能与运营商DNS、IPv6路由、跨境链路或网络缓存有关，单纯修改业务HTML无法可靠解决。

## 三、公开产物白名单

EdgeOne主入口和GitHub Pages备用入口都只能发布以下三个文件：

```text
index.html
build-manifest.json
pages-release.json
```

以下内容不得进入公开静态入口：

- `admin-*-preview.html`和其他管理员预览页面；
- 源码、测试、日志、环境变量示例和CI诊断；
- Blob管理页面或维护页面；
- 稳定版8.2.25文件；
- 任何密钥、密码、令牌或临时部署来源。

`pages-release.json`记录通道、源提交、冻结候选摘要、稳定晋升为false和正式写入为false。线上验证器以该文件为入口，重新计算HTML摘要并检查构建清单。

## 四、EdgeOne Pages主入口方案

### 4.1 已完成的仓库配置

根目录`edgeone.json`已经固定：

```text
安装命令：npm ci --ignore-scripts
构建命令：npm run edgeone:build
输出目录：./.edgeone-artifact
Node版本：22.11.0
```

构建命令会先运行完整Node门禁，再生成三文件白名单。公开响应统一配置：

```text
Cache-Control: no-cache, max-age=0, must-revalidate
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: no-referrer
```

暂不配置严格Content-Security-Policy，因为当前单HTML包含大量内联脚本和样式；未经专项兼容测试直接增加CSP可能破坏本地计算、导入导出或媒体功能。

### 4.2 将来创建预览项目时的最少人工步骤

1. 在EdgeOne Pages控制台从GitHub导入`srhovo/cloud-collab`。
2. 首次只使用非生产分支建立预览部署，不绑定正式域名，不开启任何`*_ENABLED`变量。
3. 确认构建输出目录显示为`.edgeone-artifact`，文件数为3。
4. 取得该次部署URL后执行：

```bash
npm run public:verify -- \
  --url "https://实际部署地址/" \
  --expected-commit "40位提交SHA" \
  --expected-channel edgeone-primary
```

5. 验证通过后再考虑绑定候选专用子域名；不把候选域名当作稳定版域名。

EdgeOne Pages可把自定义域名关联到生产或预览环境。选择“中国大陆可用区”或“包含中国大陆的全球可用区”时，域名需要先完成ICP备案。若没有备案，应先使用平台部署域名或不包含中国大陆的区域，不宣称已经具备中国大陆稳定覆盖。

官方参考：

- https://pages.edgeone.ai/zh/document/edgeone-json
- https://pages.edgeone.ai/document/custom-domain
- https://edgeone.ai/document/180002216676458496

## 五、GitHub Pages备用入口方案

`.github/workflows/pages.yml`已取消`main`分支自动发布，只允许手动触发。触发时必须同时输入：

```text
candidate_version = 8.2.30
confirmation = DEPLOY-CANDIDATE-8.2.30
```

工作流执行顺序：

1. 锁定依赖安装；
2. 完整Node门禁；
3. 生成`.pages-artifact`三文件白名单；
4. 上传GitHub Pages专用artifact；
5. 部署候选备用入口；
6. 用部署URL核对提交、标题、APP_VERSION、字节数、SHA-256和管理员预览页不可访问。

GitHub Pages是备用入口，不承诺中国大陆网络稳定性。自定义Actions部署需要`pages: write`、`id-token: write`和部署环境；仓库应给`github-pages`环境增加保护规则，限制为项目负责人手动执行。

官方参考：

- https://docs.github.com/zh/pages/getting-started-with-github-pages/using-custom-workflows-with-github-pages
- https://docs.github.com/en/pages/getting-started-with-github-pages/configuring-a-publishing-source-for-your-github-pages-site

## 六、自动发布预演

阶段7H新增无部署工作流`stage7h-release-rehearsal`。它会：

1. 运行`npm run release:rehearse`；
2. 重新构建8.2.30；
3. 与阶段7G冻结SHA-256比较；
4. 同时生成EdgeOne和GitHub Pages产物；
5. 比较两个通道的`index.html`和`build-manifest.json`逐字节一致；
6. 确认两个目录都只有3个普通文件；
7. 确认管理员预览页未进入产物；
8. 上传预演证据artifact。

该工作流不包含`actions/deploy-pages`，不会创建EdgeOne部署，也不会修改DNS。

## 七、线上自动验收项

真实候选部署创建后，`public:verify`检查：

- 访问地址必须为HTTPS；
- `pages-release.json`、`index.html`和`build-manifest.json`均返回200；
- HTML响应类型为`text/html`，JSON响应类型为`application/json`；
- 线上源提交等于预期提交；
- 候选版本、标题、APP_VERSION、字节数和SHA-256完全一致；
- 稳定版仍为8.2.25，晋升授权和执行均为false；
- 正式公共写入未包含；
- EdgeOne通道响应头符合仓库配置；
- `admin-sensitive-reviews-preview.html`不得返回200。

## 八、最小真实网络验收

自动化通过后，只保留无法由GitHub Actions替代的最少人工步骤：

| 设备/网络 | 操作 | 通过条件 |
|---|---|---|
| iPhone Safari + 常用Wi-Fi | 打开候选主入口并刷新一次 | 页面完整加载，版本显示8.2.30 |
| 同一iPhone + 移动数据 | 打开同一地址 | 页面完整加载，结果与Wi-Fi一致 |
| Android + 常用Wi-Fi | 打开页面并完成一次普通码单 | 本地计算和保存正常 |
| iPad或桌面浏览器 | 导入后再导出 | 本地数据不丢失，文件可下载 |
| 任一设备断网 | 已打开页面后进行本地操作 | 本地计算可继续，云端失败不破坏数据 |

发生失败时只记录：绝对时间、设备和系统、Wi-Fi或移动数据、入口URL、错误截图。不要先清空浏览器数据或删除本地HTML。

## 九、回滚

候选入口回滚不等于稳定版回滚：

- EdgeOne：把候选域名重新关联到上一条已验证部署，或删除候选域名关联；
- GitHub Pages：重新手动部署上一条已验证提交；
- 离线：继续分发阶段7G冻结的单HTML；
- 不修改稳定版8.2.25，不开启正式公共写入，不操作正式公共数据库。

任何回滚后都必须重新执行线上摘要验证，不能只以“页面能打开”为通过标准。

