- 码单器8.2.25稳定基线不得修改、替换或重新生成

- 正式公共库 `group_xiacijian / lib_xiacijian_regular` 保持版本0和空数据

## 2. 本阶段唯一主项目

在8.2.27只接收同步基础上，增加8.2.28隔离预览候选派发客户端：

1. 惰性设备注册
2. 独立本地设备凭据管理
3. `pendingCloudChanges` 队列派发
4. 幂等重试与重复响应处理
5. 错误分类与队列状态回写
6. 断网、超时和服务关闭时的零破坏降级

不开发正式公共写入、自动批准、管理员审核、删除操作或其他数据类型。

## 3. 客户端隐私边界

允许进入候选提交的字段只来自冻结的普通精确价格协议：

- groupId
- libraryId
- serviceName
- settleType（round/hour）
- unitPrice
- deviceId、submissionId及协议Hash
- 协议版本和客户端版本

不得进入候选提交、日志、错误详情或预览门禁存储的内容包括：

- 历史记录、订单、原始聊天和备注
- 最近老板、老板记忆及私人老板字段
- 锁定项、布局、模板、自定义比例和临时UI状态
- usageCount、lastUsed、timestamp、original等个人使用痕迹
- 设备令牌、预览门禁凭据、管理员凭据和其他秘密

预览门禁凭据只存在当前页面内存；刷新或关闭页面即丢失。设备令牌只保存到独立的 `cloudDeviceCredential`，不进入标准备份和旧数据迁移。

## 4. 队列和错误规则

- 本地模式：不接收、不提交
- 只接收模式：仅拉取公共更新，不生成候选
- 参与协作模式：先完成公共同步比较，再把可共享的本地普通精确价格逐条加入队列
- 同一幂等键不得对应不同请求正文
- 离线时不改变队列状态
- 429、5xx、超时和网络错误进入可重试状态
- 401清除失效本地凭据并阻断当前记录，避免无限认证循环
- 403和409按预览门禁、作用域或幂等冲突分类并阻断
- 成功或服务端幂等重复响应均回写为 acknowledged

## 5. 一次性合成对象清理

### 5.1 强制条件

- 只允许命名空间 `cloud-collab-preview-v1`
- `CLOUD_WRITE_PREVIEW_ENABLED` 必须为 `0`
- `CLOUD_PREVIEW_CLEANUP_ENABLED` 默认 `0`，执行窗口才短暂改为 `1`
- 清理密钥必须独立且不得复用预览写入密钥
- 不得把任何真实密钥、盐值、令牌、完整预览地址写入GitHub、文档、截图或聊天

### 5.2 双阶段执行

1. 调用 `inspect`：强一致列举和读取对象，只返回数量、分类和清单摘要，不返回对象Key或内容。
2. 人工核对仅包含合成对象。
3. 调用 `execute`：提交同一清单摘要及固定确认短语。
4. 删除后强一致复查对象数必须为0。

任何未知Key、正式库路径、非法结构、对象数量超限、清单变化或执行后残留都会拒绝或报错。

### 5.3 立即销毁

远程清理确认成功后，在本分支立即删除：

- `cloud-functions/api/system/cleanup-preview-fixtures-once.js`
- `src/server/preview_fixture_cleanup_once_v1.js`
- `src/server/preview_fixture_cleanup_http_v1.js`
- `src/server/preview_fixture_cleanup_runtime_v1.js`
- `tests/stage4c-preview-cleanup.test.mjs`
- `.env.example` 中的两个清理变量说明

随后重新运行完整CI，并确认仓库搜索不到清理路由和清理开关。本步骤完成前，PR保持草稿且不可合并。

## 6. 测试门禁

完整CI至少包括：

- 单文件构建和JavaScript语法检查
- 所有Node单元测试
- 冻结源SHA、旧Schema和关键业务类未变化
- 构建可重复
- 只读同步三方合并与回滚
- 设备注册、凭据持久化、幂等派发和离线行为

- 私人字段投影黑名单
- 预览门禁不持久化、设备令牌不进入正文
- 正式公共库仍为空，正式写入和自动批准仍关闭
- 一次性清理的未知对象拒绝、摘要变化拒绝和清理后复查

## 7. 回滚

在PR合并前，回滚方式为关闭PR并删除独立分支；`main`和8.2.25不受影响。

若预览部署出现异常：

1. 立即把预览写入和清理开关恢复为0；
2. 回退到main当前部署；
3. 保留本地队列，不删除用户本地数据；
4. 不对正式公共库执行任何补偿写入；
5. 通过新的独立分支修复并重新走CI和PR审核。
