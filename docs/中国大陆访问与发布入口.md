# 中国大陆访问与发布入口方案

## 1. 现象判断

“中国大陆Wi-Fi偶发打不开GitHub仓库或GitHub Pages，切移动数据或重启后恢复”更符合接入网络差异，而不是单HTML业务代码故障。常见变量包括运营商DNS、IPv6优先路由、跨境链路、缓存和GitHub静态资源路径。仅修改页面代码不能可靠消除这些因素。

## 2. 建议拓扑

| 角色 | 平台 | 用途 | 边界 |
|---|---|---|---|
| 代码权威源 | GitHub | 版本、PR、CI、审计证据 | 不把仓库页面当作面向普通用户的唯一入口 |
| 建议主入口 | EdgeOne Pages | 面向用户的静态单HTML入口 | 中国大陆节点与稳定自定义域名受域名/ICP备案条件约束 |
| 备用入口 | GitHub Pages | EdgeOne异常时的备用静态入口 | 大陆网络可用性不保证；只发布最小产物 |
| 可选源码镜像 | Gitee仓库 | 人工应急获取源码 | 当前不依赖Gitee Pages，不作为自动发布主链 |
| 离线兜底 | 已验证单HTML | 用户本机直接打开 | API未配置时安全降级，不影响本地计算和既有数据 |

这套结构不会承诺“免费、永久链接、无备案、稳定覆盖中国大陆”四项同时成立；平台公开规则并不支持这种保证。

## 3. 为什么不直接改用Gitee Pages

附件中的“GitHub Actions镜像到Gitee Pages”可以保留其源码镜像思路，但Pages服务可用性与开通状态并不适合作为当前发布前提。现阶段只考虑可选仓库镜像，不把Gitee Pages加入正式入口和CI成功条件。

## 4. EdgeOne落地顺序

在代码PR、发布证据和独立发布授权全部完成后：

1. 从GitHub导入仓库；仓库配置会用`npm ci --ignore-scripts`安装、`npm run edgeone:build`执行完整门禁，并只输出`.edgeone-artifact`中的三个公开文件。
2. 首次只创建预览部署，所有`*_ENABLED`保持`0`，私密环境变量保持空。
3. 自动核对在线标题为8.2.30，并记录源提交、字节数与SHA-256。
4. 再决定使用平台域名还是自定义域名；需要中国大陆节点时先核对ICP备案条件。
5. 真实入口确定后，再做一次最少两网络实测：同一台设备分别用常用Wi-Fi和移动数据打开主入口，确认页面加载、核心计算和离线降级。

前四步能由平台和自动化完成；只有第5步依赖用户所在的真实中国大陆网络。

## 5. GitHub Pages备用入口控制

GitHub Pages不再跟随main自动发布。每次备用入口更新都必须经过：独立手动授权、完整CI、发布账本闭环、三个文件白名单打包、线上SHA-256复核。这样即使GitHub网络偶发不可达，也不会再出现“页面能打开但仍是旧版本”的第二类故障。

## 6. 失败时的最短用户排查

在主入口上线前，不要求反复切网测试。主入口上线后若仍偶发失败，只需记录三项：失败时间、所用网络（Wi-Fi或移动数据）、页面错误现象/截图。不要先清空应用数据或删除单HTML文件；本地数据与网络连通性是两个独立问题。
