# 阶段7A：只读维护完整性核查范围冻结

## 目标

阶段7进入发布与维护，但第一步不直接开启正式环境、不增加定时写入，也不要求再次进行EdgeOne人工部署。

本阶段只建立一个可复用的服务器端只读完整性核查核心，为后续每日维护、监控、发布门禁和迁移前检查提供统一证据。

## 本阶段包含

新增`src/server/maintenance_integrity_v1.js`：

1. 复用正式普通公共事件验证器；
2. 复用正式敏感公共事件验证器；
3. 强制检查普通版本从1连续递增；
4. 强制检查敏感版本紧接普通基线连续递增；
5. 重建统一公共快照并核对：
   - groupId/libraryId；
   - baseOrdinaryVersion；
   - publicVersion/snapshotVersion；
   - `pv_<version>`游标；
   - 生成时间；
6. 检查生效记录业务键唯一；
7. 检查墓碑业务键唯一；
8. 检查同一业务键不能同时存在于生效记录和墓碑；
9. 强一致列举当前库的候选对象库存；
10. 读取候选`receivedAt`并报告无效时间数量及最老候选年龄；
11. 返回只读报告，固定声明`mutationsPerformed=0`。

## 明确排除

- 不新增EdgeOne环境变量；
- 不新增HTTP路由或管理员页面；
- 不执行Blob写入、删除或保留期清理；
- 不生成定时任务；
- 不启用正式公共写入、自动批准或管理员发布；
- 不修改用户端HTML和稳定版8.2.25；
- 不进行EdgeOne部署或人工按钮验收。

## 安全边界

- 所有Blob列举使用强一致读取；
- 每个库存前缀最多允许20,000个对象，超限失败关闭；
- 空Key、重复Key或返回其他前缀的Key失败关闭；
- 公共事件内容仍由阶段5G/6B正式验证器校验，本模块不放宽Hash、批准索引或事件结构；
- 报告只包含版本、数量、时间和布尔核查结果，不返回业务正文、设备ID、提交ID、幂等键、Blob Key或管理员秘密。

- 普通/敏感版本跳号失败；
- 快照版本或游标不一致失败；
- 重复记录、重复墓碑和记录/墓碑重叠失败；
- Blob列举越界Key失败。

现有`npm test`会自动包含本测试，不需要修改环境变量或部署配置。
