## 三、已修复的发布一致性缺陷

1. 发布候选标题与构建清单统一为8.2.30；`APP_VERSION=8.2.28`继续作为协议兼容标识。
2. 发布证据账本记录版本选择来源；版本决策与实际构建不一致时CI失败。
3. GitHub Pages取消main push自动部署，只接受单独手动授权。
4. Pages部署前重新安装锁定依赖、执行完整CI和发布审计。
5. GitHub Pages与EdgeOne公开产物均固定为三个文件，8个管理员预览页不得上传。
6. 部署后读取线上文件，复核Git提交、标题、字节数、构建清单和SHA-256。
7. 通用浏览器测试接受显式目标文件和预期版本，避免测错HTML仍显示通过。
8. README、GitHub/EdgeOne清单和中国大陆入口方案更新到阶段7F。

## 四、风险分级

| 等级 | 风险 | 控制 |
|---|---|---|
| 高 | 自动发布仓库旧`dist`导致源码、CI和线上版本不一致 | 已改为现场构建、手动授权和线上摘要复核 |
| 高 | 管理员预览页面被GitHub Pages或EdgeOne静态站点公开 | 两个平台统一使用最小产物白名单；单元测试核对文件清单 |
| 中 | 8.2.30发布版本与8.2.28内部协议标识混淆 | 构建清单同时记录两个版本并分别验证 |
| 中 | 大陆Wi-Fi无法稳定访问GitHub Pages | 主/备入口分离；EdgeOne域名和备案条件单独验收 |
| 中 | 人工清理与最终重跑证据不完整 | 继续作为机器阻断项，不伪造通过 |
| 低 | 教程视频CDN不可用 | 按需加载；核心计算与本地功能不依赖该资源 |

## 五、自动验收命令

```bash
npm ci --ignore-scripts
npm run ci
python3 tests/core_compare.py --html dist/index.html --expected-version 8.2.30
python3 tests/browser_integration.py --html dist/index.html --expected-version 8.2.30
```

`npm run ci`必须产生并验证：

- `dist/index.html`
- `dist/build-manifest.json`
- `dist/release-readiness-audit.json`
- `dist/stage7f-repository-audit.json`

`npm run pages:prepare -- --commit <40位提交SHA>`只有在发布账本无阻断时才生成`.pages-artifact`。当前有阻断时失败是正确结果。
