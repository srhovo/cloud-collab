# 阶段5G：普通共享类型缺口补齐范围冻结

## 为什么不是直接进入阶段6

总方案的阶段3要求开放普通单价、服务类型、已确认陪玩名字、新老板资料和已有老板折数下降候选；阶段6才处理区间、加价、礼物、删除和直属变化等全部人工审核内容。

当前`main@2ce00ae85564d36dcf2b97f8837cc40302f034e1`虽然已经完成普通精确价格、自动审核、管理员网页、设备治理、回滚、导出和真实联合验收，但提交协议、客户端派发和公共事件发布仍只接受`exact_price/upsert`。服务类型已随普通价格的`serviceName`共享，不需要单独数据库；尚未闭环的是：

- `playable_name`：已确认陪玩名字；
- `boss_profile`：新老板资料；
- `boss_profile`：已有老板在直属不变时的合理折数下降。

若此时直接扩展阶段6，会在普通共享基础尚不完整时同时引入删除、直属变化和复杂规则，扩大公共协议、管理员审核和客户端冲突语义的风险。因此先建立阶段5G，把总方案阶段3/4遗留的普通类型补齐，再进入阶段6敏感规则。

## 唯一主目标

在不改变现有普通精确价格行为的前提下，把公共协作系统从“只支持exact_price”扩展为支持以下普通共享类型：

1. `playable_name/upsert`；
2. `boss_profile/upsert`的新老板资料；
3. `boss_profile/upsert`的已有老板折数下降，且直属保持不变、折数在0.8至1之间。

服务类型继续作为`exact_price.payload.serviceName`共享，不新增重复的`service_type`数据库。

## 数据白名单

### playable_name

严格字段：

```json
{
  "name": "最终确认的陪玩名字"
}
```

规则：

- Unicode NFKC规范化；
- 清理首尾空格与连续空格；
- 长度1至30个字符；
- 禁止控制字符、URL、邮箱、明显联系方式；
- 不上传原始聊天、订单来源、错误名字、修正上下文、时间或本地ID。

业务键：`groupId + normalizedName`。陪玩名字不属于单一价格库内容，但V1现有提交信封仍保留`libraryId`作为绑定与权限作用域；服务端业务键不得把libraryId作为名字去重键。

### boss_profile

严格字段：

```json
{
  "bossName": "老板名",
  "paiDan": "直属/派单",
  "discount": 0.97
}
```

规则：

- 老板名与直属分别NFKC规范化并清理空格；
- 老板名长度1至30；直属长度0至30；
- 折数必须为0.8至1之间的有限数，最多4位小数；
- 禁止联系方式、控制字符和本地字段；
- `bossId`为服务器身份，不从本地老板对象payload读取；
- 新老板临时业务键采用`groupId + normalizedBossName`，同名歧义进入管理员审核，不自动猜测；
- 公共事件发布后由服务器保存稳定bossId与名字映射。

## 普通审核规则

### 陪玩名字

- 两个不同设备提交相同规范名字：可自动批准；
- 可信设备提交一次：可自动批准；
- 已存在相同名字：幂等/已存在，不重复发布；
- 同名规范化异常、内容风险或身份冲突：进入审核或拒绝；
- 名字永不携带订单和聊天证据。

### 新老板

- 两个不同设备提交相同老板名、直属和折数：可自动批准；
- 可信设备提交一次：可自动批准；
- 同名但直属或折数不同：进入管理员审核；
- 本地没有该老板的同团用户可以通过公共同步新增。

### 已有老板折数下降

只有同时满足下列条件才属于普通候选：

- 可唯一解析到同一公共bossId；
- `paiDan`与当前公共值完全一致；
- 新折数小于当前折数；
- 新旧折数都在0.8至1；
- 下降幅度不触发异常阈值；
- 两个不同设备提交相同新值，或可信设备提交一次；
- 没有其他冲突候选。

否则一律进入人工审核或按稳定错误码拒绝，不得自动覆盖。

## 明确排除并留给阶段6

- 老板直属变化；
- 老板折数升高；
- 删除老板、删除价格或删除其他规则；
- 区间规则、加价规则、礼物规则；
- 整库覆盖、库重命名、重新绑定；
- 批量导入转公共提交；
- 正式公共库写入和正式用户启用。

## 技术拆分

阶段5G按一个正式PR推进，但实现顺序锁定为：

1. **多类型协议底座**：按dataType分派payload规范化、业务键和内容Hash；现有exact_price结果必须保持兼容；
2. **候选接收与审核决策**：所有类型沿用不可变提交、幂等、设备认证和限流；
3. **公共事件与快照多类型化**：事件schema保持向后兼容，快照能同时投影价格、名字和老板；
4. **管理员队列与详情**：按类型脱敏显示旧值、新值、冲突与影响范围；
5. **客户端投影与显式入队**：只监听用户明确保存/确认操作，不监听通用Storage写入；
6. **同步合并**：公共名字/老板新增不覆盖本地修改，冲突保留本地值；
7. **测试与静态门禁**。

若公共事件schema必须升级，必须先证明旧schemaVersion 1事件继续可读，旧exact_price客户端不会停止同步；不得用“顺便重构”扩大范围。

## 默认关闭与隔离

新增独立预览门禁并默认关闭：

```text
CLOUD_ORDINARY_TYPES_PREVIEW_ENABLED=0
```

要求：

- Blob固定为`cloud-collab-preview-v1`；
- group/library固定为`group_fixture / lib_receive_fixture`；
- 正式公共写入关闭；
- 阶段5D、5E、5F管理员能力默认关闭；
- 新类型在服务器、客户端和静态门禁三层均需独立开关；
- 关闭时现有exact_price链路行为不变。

## 客户端边界

- 候选版本继续在现有8.2.28开发线演进，暂不改稳定8.2.25；
- 只有绑定模式为`collaborate`且来源为`user`的显式操作可以入队；
- `initialBinding`必须由用户明确选择，并拆分为原子候选；
- `import`、`migration`、`cloudPull`、`rollback`和`system`永不入队；
- 自动记忆、订单使用、最近老板、历史和聊天不得产生云提交；
- 私人字段过滤后必须重新构造新对象，不能直接序列化本地对象。

## 最低验证门槛

- exact_price协议、Hash、自动批准、公共事件、回滚和导出回归字节/语义兼容；
- playable_name与boss_profile严格字段白名单和递归禁用字段测试；
- 名字NFKC、空格、长度、链接、邮箱和联系方式边界；
- 老板折数0.8至1、直属不变、下降/升高分流和同名歧义；
- 两设备确认、可信设备一次批准、冲突进入审核；
- 不同设备、令牌和提交身份校验；
- 幂等重放与同键不同正文冲突；
- 多类型公共事件、增量和最新快照一致性；
- 管理员批准、拒绝、修改后批准及审计；
- 客户端只在显式用户操作入队，系统/导入/拉取不入队；
- 云端失败不影响本地保存和码单；
- 私人字段扫描、普通备份schema保持不变；
- Node全量、静态隐私门禁、冻结核心计算、普通用户和管理员Chromium回归。
