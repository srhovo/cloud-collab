# 阶段4D：普通精确价格自动审核与公共事件基础

## 1. 本阶段定位

本阶段只建立服务端自动审核与公共事件的核心状态机，不把该能力接入现有公网提交路由，也不修改用户端页面、EdgeOne环境变量或现有只读接口。

当前 `main` 的8.2.28客户端仍只向隔离候选区提交；现有提交HTTP响应继续固定返回：

```text
publicMutationAllowed=false
autoApprovalEnabled=false
```

因此，合并本基础后也不会自动改变线上行为。下一阶段必须另建独立分支，经过隔离预览和两台真实设备验收后，才允许讨论路由接入。

## 2. 支持的审核规则

仅处理普通精确价格：

```text
dataType=exact_price
operation=upsert
groupId=group_fixture
libraryId=lib_receive_fixture
```

审核结果：

1. 首个普通设备提交：`waiting_confirmation`；
2. 两个不同设备提交相同规范化内容：`auto_approved`；
3. 同一设备重复提交：仍只计一个独立确认；
4. 可信设备提交一次：`auto_approved`；
5. 同一业务键出现不同候选值：`pending_review`；
6. 候选值与现有公共值不同：`pending_review`；
7. 候选值已与公共值相同：`duplicate_noop`，不新增公共事件。

本阶段不处理名字、老板、区间规则、加价、礼物、删除及其他敏感操作。

## 3. Blob对象结构

### 3.1 独立设备确认

```text
submissions/{libraryId}/matches/{businessKey}/{contentHash}/{deviceId}.json
```

设备ID位于Key中，因此同一设备无论重复提交多少次，同一内容只计一次确认。

### 3.2 可信设备标记

```text
devices/trusted/{deviceId}.json
```

可信设备记录必须明确包含 `trusted=true`，且不存在撤销时间。此基础不提供创建或修改可信设备的公网接口。

### 3.3 待人工审核标记

```text
reviews/{libraryId}/pending/{businessKey}/{contentHash}.json
```

冲突候选各自保留，不覆盖公共数据。记录只保存公共审核所需投影，不包含历史、订单、聊天、备注、布局、最近老板、设备令牌或其他私人字段。

### 3.4 不可变批准索引

```text
public/{libraryId}/approvals/{approvalId}.json
```

`approvalId`由价格库、业务键和内容Hash确定性生成。同一内容重复批准只能命中同一索引。

### 3.5 只增公共事件

```text
public/{libraryId}/events/{12位版本号}.json
```

公共事件包含批准模式、参与确认的设备ID、业务键、内容Hash和白名单价格投影。事件不可覆盖。

### 3.6 版本快照

```text
public/{libraryId}/snapshots/{12位版本号}.json
```

快照只由有效公共事件重建。有效事件必须同时满足：

- 事件结构合法；
- 对应批准索引存在；
- 批准索引明确指向该事件Key及版本。

## 4. 并发模型

多个请求可能同时预留公共事件版本。为了避免竞争请求生成两个有效批准：

- 每个批准内容使用唯一确定性 `approvalId`；
- 同一 `approvalId` 若预留多个事件槽，批准索引固定选择最小版本事件；
- 没有被批准索引指向的竞争事件视为孤立槽，不进入快照和公共变化；
- 不同批准内容继续使用只增版本槽，公共版本保持单调增加。

Blob `onlyIfNew` 在预读为空、实际写入时被另一请求抢占的情况下，仓储层会强一致复读：

- 对象已出现：统一归类为 `BLOB_ALREADY_EXISTS`，允许上层按竞争处理；
- 对象仍不存在：保留为真实 `BLOB_ONLY_IF_NEW_FAILED`；
- 复读失败：不得误判为竞争成功。

## 5. 失败恢复

### 5.1 事件写入失败

若公共事件未成功写入：

- 不创建批准索引；
- 不生成快照；
- 重放同一候选可重新预留事件并完成批准。

### 5.2 批准索引竞争

若另一个请求已经写入同一批准索引：

- 强一致读取现有索引；
- 使用该索引指向的唯一有效事件；
- 不产生第二个有效批准。

### 5.3 快照写入失败

事件和批准索引已经有效、但快照写入失败时：

- 公共事件仍保持不可变有效；
- 重放相同候选会按公共重复处理；
- 同时重新生成并补写缺失快照；
- 不新增事件版本。

## 6. 强制安全边界

- 不修改8.2.25稳定文件；
- 不修改8.2.28客户端行为；
- 不接入 `/api/submissions/create`；
- 不新增自动批准环境开关或密钥；
- 不开放正式公共库写入；
- 不开放管理员审核、可信设备管理或人工批准接口；
- 不修改现有静态只读目录；
- 所有审核、事件、索引和快照读取均使用强一致Blob读取；
- 自动审核核心文件不得被打包进用户HTML。

## 7. 测试覆盖

专项测试覆盖：

- 首设备等待；
- 两设备一致；
- 同设备重复去重；
- 可信设备一次批准；
- 候选冲突；
- 公共值冲突；
- 公共重复无操作；
- 孤立事件过滤；
- 同批准并发竞争；
- 多批准版本递增；
- 事件写失败重放；
- 快照写失败重放；
- Blob `onlyIfNew` 平台竞争识别；
- 公网路由、环境变量与客户端构建保持未接入状态。

## 8. 下一阶段门禁

完成本基础并通过完整CI后，下一阶段才可建立“隔离预览路由接入”分支。该阶段至少需要：

1. 独立的自动审核预览开关；
2. 继续硬锁 `group_fixture / lib_receive_fixture`；
3. 动态公共版本、增量和快照只服务于隔离测试库；
4. 两台真实设备提交相同值后只生成一个有效批准事件；
5. 冲突值必须进入审核且公共版本不变；
6. 测试完成后清理合成对象并撤除预览部署；
7. 未经单独授权，不得启用正式公共写入。
