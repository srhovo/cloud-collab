# 阶段4D：普通精确价格自动审核与公共事件基础

## 阶段定位

本阶段只建立服务端审核状态机，不接入现有公网提交路由，不修改8.2.28客户端、EdgeOne配置或只读接口。现有提交响应继续保持：

```text
publicMutationAllowed=false
autoApprovalEnabled=false
```

因此合并本基础不会改变线上行为。

## 普通精确价格规则

仅处理 `exact_price / upsert`。

### 新增价格

- 第一个普通设备：等待第二设备；
- 两个不同设备提交相同规范化内容：自动批准；
- 同一设备重复提交：同一轮只计一次；
- 可信设备提交一次：可自动批准；
- 同一业务键出现不同值：进入人工审核；
- 内容已与公共值相同：按无操作成功，不新增事件。

### 修改已有价格

必须同时满足以下条件才自动批准：

- 两个不同设备提交相同新值；
- 业务键与现有项目相同；
- 涨跌幅不超过±10%；
- 当前公共基线未变化；
- 同一轮没有其他冲突值。

恰好±10%允许自动批准；超过±10%进入审核。可信设备不能单独批准已有价格修改。

## 分轮确认

确认对象按业务键和当前公共记录版本分轮：

```text
submissions/{libraryId}/matches/{businessKey}/pv_{基线版本}/{contentHash}/{deviceId}.json
```

新增项目使用基线版本0。空公共基线只允许 `approvedVersion=0` 且 `contentHash/unitPrice` 同时为 `null`；其他版本仍必须是有效正式记录。公共记录版本增加后自动进入新一轮，历史确认不参与后续改价。

待审核标记同样按基线版本隔离：

```text
reviews/{libraryId}/pending/{businessKey}/pv_{基线版本}/{contentHash}.json
```

## 批准与公共数据结构

```text
devices/trusted/{deviceId}.json
public/{libraryId}/transitions/{businessKey}/{基线版本}.json
public/{libraryId}/approvals/{approvalId}.json
public/{libraryId}/events/{12位版本号}.json
public/{libraryId}/snapshots/{12位版本号}.json
```

- `transition`保证同一业务键的同一公共基线只能采用一个目标值；
- `approvalId`包含基线版本、基线内容和目标内容，价格改回历史值时仍生成新事件；
- 公共事件不可变，只增不改；
- 快照只由批准索引明确采用的事件重建；
- 并发产生但未被索引采用的孤立事件安全忽略；
- 对象数量超过安全上限时中止自动审核。

## 并发与恢复

- 条件写入被其他请求抢占后，使用强一致读取区分正常竞争和真实存储失败；
- 同目标值竞争复用已采用结果；
- 同基线不同目标值竞争进入审核；
- 公共基线变化后，旧候选不能继续发布；
- 事件写失败时不创建批准索引和快照；
- 快照写失败时保留有效事件，重放补写快照且不新增有效事件。

## 安全边界

- 不修改8.2.25稳定文件；
- 不接入现有提交路由；
- 不开放正式公共库写入；
- 不开放管理员、可信设备管理或人工批准接口；
- 不处理名字、老板、区间、加价、礼物和删除；
- 自动审核服务端文件不得打包进用户HTML；
- 所有审核、事件、索引和快照读取使用强一致Blob读取。
