# 阶段7P：正式设备注册与精确价格候选入队

## 目标

在不部署EdgeOne、不访问真实Blob、不生成真实密钥和不晋升稳定版的前提下，完成正式公共协作写入链的第一小步：设备注册、设备令牌鉴权、精确价格候选入队和生产限流。

本阶段只接收候选，不自动修改公共库。普通自动审核仍由下一阶段单独接入。

## 公共路由与模式分发

继续复用既有客户端路径：

```text
POST /api/device/register
POST /api/submissions/create
```

分发规则：

```text
CLOUD_PRODUCTION_ENABLED=1  → 正式处理器
CLOUD_PRODUCTION_ENABLED=0或未配置 → 原隔离预览处理器
其他值 → 503失败关闭
```

因此既有阶段4B.2预览链路和浏览器验收无需改路径；只有生产总开关明确为1时才进入正式逻辑。

## 正式配置边界

正式设备注册和候选入队要求：

- 生产总开关开启；
- 只读同步已开启；
- 普通提交已开启；
- `see / see_cz`稳定映射为`group_see / lib_see_cz`；
- 公共Store固定为`cloud-collab-production-v1`；
- 精确匹配`CLOUD_PRODUCTION_PUBLIC_ORIGIN`；
- `X-Cloud-Collab-Access-Key`使用独立生产访问密钥；
- 生产限流盐值与访问密钥不同；
- 设备令牌由既有设备注册和鉴权核心生成、校验；
- 所有密钥只存在于EdgeOne私密环境变量。

## 设备注册

正式设备注册复用既有`registerDevice`核心，并增加生产限流：

```text
限流窗口：60秒
限流主体：deviceId的加盐SHA-256
Blob Key前缀：production-rate/device-register/
```

限流Key不包含设备ID、盐值或访问密钥明文。

HTTP处理顺序：

1. 校验生产运行时和分阶段开关；
2. 校验访问密钥；
3. 校验JSON正文；
4. 创建指定生产Store；
5. 消耗限流槽；
6. 调用设备注册核心；
7. 返回一次性设备令牌。

配置关闭或访问密钥错误时，在解析正文和创建Blob Store之前阻断。

## 精确价格候选入队

正式精确价格提交复用：

- `normalizeSubmission`；
- 业务Key和内容Hash协议；
- `authenticateDevice`；
- `acceptSubmission`；
- 不可变候选和幂等索引；
- `pendingSubmissionKey`。

附加约束：

- 提交必须使用协议ID`group_see / lib_see_cz`；
- Authorization设备必须与正文deviceId一致；
- 新候选每台设备5秒一个限流槽；
- 精确幂等重放绕过新的限流槽；
- 相同幂等键但不同正文由既有接收核心拒绝；
- 候选入队后`publicMutationAllowed=false`；
- `autoApprovalEnabled=false`；
- `stablePromotionAuthorized=false`。

## 自动审核边界

阶段7P不接入自动审核处理器。如果环境中提前开启：

```text
CLOUD_PRODUCTION_AUTO_APPROVAL_ENABLED=1
```

当前正式提交路由返回503：

```text
PRODUCTION_AUTO_APPROVAL_HANDLER_REQUIRED
```

并且不会创建Blob Store。这样防止“配置宣称已自动审核，但实际仍只入队”的错误运行状态。

## CORS和HTTP边界

- 仅允许`POST`和受限`OPTIONS`；
- 只对精确配置的正式来源返回CORS；
- 禁止`Access-Control-Allow-Origin: *`；
- 仅允许`Content-Type`、`Authorization`和`X-Cloud-Collab-Access-Key`；
- 响应`Cache-Control: no-store`；
- 服务器错误不返回内部详情或密钥；
- 429返回有限`Retry-After`。
