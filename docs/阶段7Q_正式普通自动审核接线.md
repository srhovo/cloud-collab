# 阶段7Q：正式普通自动审核接线

## 目标

在不部署EdgeOne、不访问真实Blob、不生成真实密钥、不晋升稳定版的前提下，把阶段4D已经通过自动化与真实链路验证的普通精确价格自动审核引擎接入正式提交路由。

## 接线范围

正式入口继续使用：

```text
POST /api/submissions/create
```

运行时要求以下开关按顺序全部满足：

```text
CLOUD_PRODUCTION_ENABLED=1
CLOUD_PRODUCTION_READ_SYNC_ENABLED=1
CLOUD_PRODUCTION_ORDINARY_SUBMISSION_ENABLED=1
CLOUD_PRODUCTION_AUTO_APPROVAL_ENABLED=1
```

当自动审核关闭时，行为保持阶段7P不变：只接收不可变候选，返回等待状态，不修改公共库。

当自动审核开启时，正式运行时按以下顺序执行：

1. 校验生产配置、精确来源访问密钥、协议作用域和设备身份；
2. 对新候选消耗生产提交限流槽，精确幂等重放绕过新限流槽；
3. 调用既有`acceptSubmission`写入不可变候选；
4. 对候选进行强一致读回；
5. 调用既有`reviewExactPriceCandidate`；
6. 根据可信设备、两个不同设备相同确认、冲突候选和公共基线决定等待、人工复核或自动批准；
7. 自动批准时仍使用既有不可变公共事件、批准索引、迁移索引和快照协议。

## 响应语义

```text
autoApprovalEnabled=true
publicMutationAllowed=true
publicMutationApplied=true|false
autoApprovalResult.status=waiting_confirmation|pending_review|auto_approved
stablePromotionAuthorized=false
```

`publicMutationAllowed`表示当前运行时允许普通自动审核发布公共事件；`publicMutationApplied`只表示本次请求是否实际创建了新的公共事件。精确重放或公共值已相同时，允许能力仍开启，但本次修改为`false`。

## 故障恢复

候选入队成功而审核阶段暂时失败时，不删除候选、不伪造审核成功。客户端使用同一幂等请求重试时：

- 已存在候选不会再次消耗生产提交限流槽；
- 候选正文必须与原请求一致；
- 自动审核重新执行；
- 既有批准索引和公共事件保证不会产生第二个公共版本。

候选接收器若声称成功但强一致读回不到候选，返回：

```text
PRODUCTION_CANDIDATE_NOT_FOUND_AFTER_ACCEPT
```

并且不调用自动审核引擎。

## 复用与边界

本阶段直接复用：

- `submission_acceptance_v1.js`；
- `auto_approval_engine_v1.js`；
- `blob_repository_v1.js`；
- 阶段4D已经验证的可信设备、双设备确认、冲突转人工、不可变事件和幂等协议。

本阶段不实现：

- 普通陪玩名字和老板资料的正式提交；
- 正式管理员登录、队列、批准、拒绝或编辑后批准；
- 正式敏感提交；
- EdgeOne部署和环境变量写入；
- 真实Blob初始化或公共数据修改；
- 候选HTML修改；
- 稳定版晋升。

## 自动化验证

专项测试覆盖：

- 普通设备首次提交等待第二台确认；
- 可信设备首次提交自动批准；
- 两个不同设备提交相同内容只产生一个公共事件；
- 精确重放不产生第二个公共事件；
- 审核暂时失败后使用同一幂等请求恢复；
- 候选未持久化时失败关闭；
- HTTP在等待时返回202、自动批准完成时返回200；
- 精确CORS、默认关闭开关、无新增审核密钥和稳定晋升继续关闭。

所有测试仅使用内存Store。真实EdgeOne Blob读、写、删均为0。
