# 阶段7E：发布收口基线审计

## 一、目的

阶段7A至7D已经完成只读完整性核查、每日维护计划、每日快照、迁移导出和默认关闭维护执行核心。阶段7E不继续增加业务能力，而是建立发布收口的机器可读证据账本和自动审计门禁。

本阶段不会晋升版本，不会改名候选文件，不会部署EdgeOne，不会写入或删除Blob，也不会开启正式公共写入。

## 二、自动核查范围

`npm run release:audit`会在完成候选构建后核查：

- 稳定基线`码单器8.2.25.html`存在且标题版本正确；
- 当前兼容候选仍为`8.2.28`，构建清单、文件大小和SHA-256一致；
- 构建入口继续使用既有兼容候选构建器；
- `.env.example`中的全部`*_ENABLED`能力默认值均为`0`；
- 示例环境中密码、密钥、会话秘密和盐值均为空；
- API来源和管理员公开来源未固定到临时部署；
- 自动化、真实设备、清理和临时资源证据分区完整；
- 审计本身不修改稳定基线、候选、Blob或部署。

审计报告写入构建目录：

```text
dist/release-readiness-audit.json
```

该文件是CI生成物，不作为稳定版或候选版晋升结果。

## 三、当前真实结论

当前状态固定为`decision_required`，原因不是代码或CI失败，而是仍有项目负责人决策和发布前管理证据未闭环：

1. 版本号尚未决定。当前兼容外壳为8.2.28；总方案建议完整边界、回滚、迁移和实机测试候选采用8.2.30，但版本号变化必须由项目负责人明确决定。
2. 最终干净数据下的完整快照与墓碑人工重跑因人工成本终止。已完成的真实链路证据继续保留，但是否接受该豁免必须单独确认。
3. 清理仅有用户“已按流程完成”的报告，没有记录精确删除数量和两次独立零对象数字。
4. 临时EdgeOne资源和聊天中出现过的临时凭据仍需在发布前确认已删除或轮换。

这些项目不会被自动伪装成“已通过”。

## 五、失败关闭条件

以下任一情况会使CI失败：

- 稳定基线或候选文件缺失；
- 标题版本与证据账本不一致；
- 构建清单SHA-256或字节数与候选不一致；
- 兼容构建契约被绕过；
- 任一预览或写入能力在`.env.example`中默认开启；
- 示例环境包含非空凭据、盐值或临时部署来源；
- 自动化证据不是success/passed；
- 发布策略允许自动晋升、自动开启正式写入或跳过项目负责人授权。
