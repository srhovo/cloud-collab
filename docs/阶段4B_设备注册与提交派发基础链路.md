# 阶段4B：设备注册与提交派发基础链路

## 目标

在不修改公共快照、不自动批准、不开放管理员写入的前提下，建立：

- 匿名设备注册
- HMAC设备令牌
- Blob原子幂等占位
- 普通精确价格候选持久化
- 单设备分钟/小时限流
- 客户端队列派发器
- 网络失败重试与永久错误阻断

## 路由

```text
POST /api/v1/device-register
POST /api/v1/submissions
```

两个路由均位于 `cloud-functions/`，使用Node.js运行时。没有显式环境开关时返回503；没有安全签名密钥或Blob存储时同样失败关闭。

## 环境变量

```text
CLOUD_COLLAB_DEVICE_REGISTRATION_ENABLED=false
CLOUD_COLLAB_SUBMISSION_INTAKE_ENABLED=false
CLOUD_COLLAB_DEVICE_TOKEN_SECRET=
CLOUD_COLLAB_BLOB_STORE=cloud-collab-v1
CLOUD_COLLAB_DEVICE_TOKEN_TTL_MS=2592000000
CLOUD_COLLAB_SUBMISSION_MINUTE_LIMIT=20
CLOUD_COLLAB_SUBMISSION_HOUR_LIMIT=200
```

真实 `CLOUD_COLLAB_DEVICE_TOKEN_SECRET` 只能配置在EdgeOne项目环境变量中，最少32字节，禁止提交到仓库或写入HTML。

## 设备令牌

令牌格式：

```text
dvt1.<base64url payload>.<HMAC-SHA256 signature>
```

令牌只允许出现在 `Authorization: Bearer ...` 请求头和本地 `cloudDeviceCredential` 中；不得进入提交正文、待上传队列、错误详情、普通备份或公共导出。

服务器设备档案只保存tokenVersion、tokenId、状态和时间，不保存明文设备令牌。

## 原子幂等

阶段4B使用Blob的 `onlyIfNew` 创建幂等占位：

1. 服务端按阶段4A冻结协议重新计算三个Hash。
2. 使用 `idempotencyKey` 的SHA-256摘要作为Blob路径。
3. `onlyIfNew` 成功后才进入候选保存流程。
4. 同一幂等键、相同正文返回原结果。
5. 同一幂等键、不同正文返回409。
6. 任何候选都保持 `publicMutationAllowed:false` 和 `autoApprovalEnabled:false`。

## 候选状态

当前只产生：

```text
waiting_confirmation
```

两设备匹配、可信设备资格、审核与正式公共版本发布在后续阶段接入。阶段4B不会改写 `catalog.js`、公共快照、增量事件或publicVersion。

## 客户端派发器

`src/cloud_collab_submission_client.js` 已提供独立、尚未嵌入8.2.27页面的派发模块：

- 凭据缺失时惰性注册
- 令牌只放Authorization头
- `queued/retry_wait → sending → acknowledged`
- 网络、超时、429和5xx进入 `retry_wait`
- 400/401/403/幂等正文冲突等永久错误进入 `blocked`
- 启动和正常码单不等待派发结果

本阶段只验证模块和接口底座。8.2.28候选页面的用户语义入队钩子与自动派发开关在下一子阶段接入。

## 安全结论

- 默认部署状态仍然不可写。
- 开关、密钥或Blob任一缺失都不会假成功。
- 不上传历史、订单、聊天、备注、金额、布局、自定义比例、使用统计或本地时间。
- 不监听底层localStorage写入。
- 不把令牌放入日志、UI状态或提交正文。
- 阶段3B只接收同步与8.2.25稳定基线保持不变。
