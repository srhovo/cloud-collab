# 阶段7X：设备治理独立门禁

## 背景

阶段7W已接入正式设备治理，但原实现复用`CLOUD_PRODUCTION_ADMIN_REVIEW_ENABLED`。这会产生两个运维问题：

- 开启人工审核会同时开放设备治理，无法单独暂停；
- 关闭人工审核会同时失去设备状态维护能力。

阶段7X只解耦门禁，不修改六个API、设备引用、不可变事件、审计、幂等或认证语义。

## 新开关

```text
CLOUD_PRODUCTION_DEVICE_GOVERNANCE_ENABLED=0
```

规则：

- 模板显式默认0；
- 旧环境缺失时按0处理；
- 开启时必须同时开启生产总开关和正式管理员身份；
- 不依赖普通/敏感审核开关；
- 开启时要求`CLOUD_ADMIN_DEVICE_REF_SALT`；
- 初始化模式要求该开关保持0。

## 独立生命周期

```text
人工审核关闭 + 设备治理开启 → 设备治理可用
人工审核开启 + 设备治理关闭 → 审核可用，设备治理返回503
生产总开关关闭               → 两者都不可用
管理员身份关闭               → 两者都不可用
```

模式分发仍由`CLOUD_PRODUCTION_ENABLED`决定，因此生产子开关关闭不会回退隔离预览处理器。

## 保持不变

- `GET /api/admin/devices`
- `GET /api/admin/devices/detail`
- 可信状态维护与设备状态维护四个POST入口
- 正式生产issuer会话
- 生产公共Blob
- 不可逆`deviceRef`
- 同源限制
- 不可变治理事件、请求索引和审计
- 精确重放和正文冲突检测
- 设备状态对认证的即时影响
- `stablePromotionAuthorized=false`

## 验证

阶段7W专项测试更新为：

- 缺失新开关时为false；
- 模板值必须为0；
- 开启设备治理但关闭审核时仍可读取正式配置；
- 关闭设备治理时单独返回503；
- 开启设备治理但关闭管理员身份时配置失败；
- 正式处理器不再读取`adminReview`；
- 阶段5D、7W、7T和生产运行时回归继续通过。
