# 阶段7D：默认关闭维护执行核心范围冻结

## 一、阶段目标

阶段7A已经建立只读完整性核查，阶段7B已经建立每日维护计划，阶段7C已经建立每日快照与迁移导出包。本阶段把三者组合成一个默认关闭、显式执行、失败关闭的维护执行核心。

本阶段仍不部署EdgeOne，不新增HTTP路由，不创建定时任务，不写Blob，不保存维护标记，也不要求人工按钮验收。

## 二、执行模式

执行核心只有两种模式：

1. `plan_only`
   - 默认模式；
   - 只执行完整性核查和维护计划；
   - 不生成快照或迁移包；
   - 不调用任何工件构建器。

2. `build_due_artifacts`
   - 必须由调用方明确指定；
   - 只构建维护计划中已经判定到期的工件；
   - 快照未到期则不构建快照；
   - 迁移导出未到期则不生成ZIP；
   - 不执行持久化。

任何其他模式都在读取公共数据之前失败关闭。

## 三、执行链路

```text
阶段7A完整性核查
→ 阶段7B维护计划
→ 判断每日快照与迁移导出是否到期
→ 显式模式下只构建到期工件
→ 校验工件作用域、公共版本、摘要和零变更标记
→ 输出工件字节与建议维护标记
```

执行结果固定包含：

- 作用域；
- 执行时间；
- 公共版本锁；
- 完整性报告；
- 维护计划；
- 已构建动作；
- 延后或需人工处理的动作；
- 工件字节与摘要；
- 建议写入的维护标记；
- `mutationsPerformed=0`；
- `automaticPersistenceAllowed=false`。

## 四、一致性保护

- 完整性报告必须全绿、只读且零变更；
- 维护计划作用域与公共版本必须和报告一致；
- 每日快照工件与迁移包必须使用同一公共版本；
- 两个工件同时生成时，独立快照Hash必须与迁移包内嵌快照Hash一致；
- 执行期间公共数据发生变化导致版本或Hash漂移时返回稳定冲突，不输出可接受结果；
- 工件生成器若报告执行过任何变更，执行核心拒绝接收。

## 五、候选库存告警

候选库存告警不会阻止只读公共快照和迁移包生成，但会：

- 把执行状态标记为`attention_required`；
- 保留`review_candidate_inventory`等人工动作；
- 不自动删除、修复、批准或拒绝候选。

## 六、安全边界

- 新增环境变量：0；
- 新增HTTP路由：0；
- 新增管理员页面：0；
- 定时任务：0；
- Blob写入或删除：0；
- 自动保存快照、ZIP或维护标记：0；
- 正式公共写入、自动批准和管理员发布：继续关闭；
- 普通用户HTML与稳定版`码单器8.2.25`：不修改。

## 七、自动化验证

专项Node测试覆盖：

- 默认`plan_only`不调用工件生成器；
- 显式模式构建两个到期工件；
- 标记新鲜时零工件生成；
- 只构建单个实际到期工件；
- 快照Hash漂移失败；
- 工件公共版本漂移失败；
- 工件报告执行过变更时失败；
- 候选库存告警保留人工动作；
- 无效模式在任何读取前失败；
- 底层完整性读取异常转换为稳定错误。

现有完整CI继续覆盖冻结核心、普通用户页面及阶段5A至6B全部浏览器回归。

## 八、后续

阶段7D合并后，剩余发布与维护工作应转向发布收口，而不是继续无限增加内部模块。下一阶段建议建立默认关闭的管理员维护下载入口或CI维护工件入口，随后进行候选版晋升评估。入口仍应优先通过自动化完成，人工操作只保留平台权限和最终发布确认。
